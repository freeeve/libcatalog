## The structural fix: lockstep the three modules (Eve's ask)

The repo ships three Go modules with three independent tag families that
have already drifted into colliding meanings:

    root     v0.9.0            (schema emitter)
    hugo/    v0.8.0 pushed, v0.9.0 local-only
    backend/ its own sequence

Yet the modules are tightly coupled -- the projection schema version binds
root<->hugo on every schema bump, and backend consumes both. Consumers
currently maintain pairing tables by hand (queerbooks tasks/002: "pins hugo
module v0.7.0 and lcat-side libcatalog v0.8.0 together" -- awkward and
already stale twice).

Proposal:

- **Every release tags all three modules with the same version at the same
  commit** (v0.10.0 + hugo/v0.10.0 + backend/v0.10.0), pushed together.
- A `scripts/release.sh <version>` that tags the triple and pushes
  atomically -- removing the by-hand step this task exists because of.
- Unchanged modules still get the bump: no-change tags are cheap on the Go
  proxy, and "one number everywhere" is the point -- a consumer pins
  v0.10.0 across the board and compatibility is guaranteed by construction.
- Since v0.9.0 is burned on mismatched content, **start the lockstep at
  v0.10.0** at (or after) 33ad1e5.
- Document in README/ARCHITECTURE: the version number IS the compatibility
  contract (schema version maps 1:1 to a release version).

queerbooks will pin all its libcatalog deps to the lockstep number once
shipped.
